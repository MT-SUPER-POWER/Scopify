# 工程化提示词模板 — 使用指南

> 一套面向 **AI Agent（Claude）** 的结构化需求模板，帮助你在提需求、改 Bug、审查代码等场景下一次性把问题说清楚，减少来回澄清的成本。

---

## 🧭 什么时候用哪个模板

```
你的需求是什么？
│
├─ 要加新功能 / 新页面 / 新模块
│   └─ → 01-feature-request.md
│
├─ 发现现有功能不正常 / 报错了
│   └─ → 02-bug-report.md
│
├─ 写完了代码，想让 AI 审查一轮
│   └─ → 03-code-review.md
│

│   └─ → 04-refactoring.md
│
├─ 想为现有功能补充单元测试 / 集成测试
│   └─ → 05-test-writing.md
│
└─ 页面卡顿 / 加载慢 / 内存占用高
    └─ → 06-performance-optimization.md
```

## ✍️ 填写原则

### 1. "足够少，足够多"

- **足够少**：只写 AI 需要知道的，不写背景故事
- **足够多**：把"做了这个会不会影响 X"、"边界情况 Y" 这种你想到但没写的问题写进去

### 2. 不确定的字段不空着

拿不准的地方写 **「不确定：具体是…」**，AI 会帮你确认或给出方案。

### 3. 可选的标了「(可选)」就别强填

每个模板的字段分两级：

| 标记 | 含义 |
|------|------|
| 无标记 | **必填** — 缺少这些 AI 无法准确理解需求 |
| (可选) | **选填** — 有就填，没有可以跳过 |

### 4. 模板可以裁剪

用的时候直接复制 .md 文件里 `---template-start---` 到 `---template-end---` 之间的内容，粘贴到对话里填写。

```markdown
<!-- 1. 复制对应模板的 template 区内容 -->
<!-- 2. 在对话开头粘贴并填入具体信息 -->
<!-- 3. 直接发送，Claude 会按模板结构理解你的需求 -->
```

> 用 feature-request 模板帮我实现下面的需求：

> ## 背景与动机
> 目前歌手页没有热门歌曲列表，用户想看歌手的热门歌曲需要跳到搜索…
>

## ⚠️ 常见误区

| ❌ 误区 | ✅ 正确做法 |
|---------|------------|
| 只写两三行让 AI 自己猜 | 按模板字段提供上下文，AI 给出的方案更准确 |
| 把所有背景故事全写进去 | 只写跟实现直接相关的信息 |
| 不说验收标准 | 明确「做成什么样算好」避免反复改 |
| 把多个需求放一起 | 一个需求一张卡，复杂的拆成多个 |
| 不指定技术范围 | 告诉 AI 改哪些文件/模块，减少无效探索 |
